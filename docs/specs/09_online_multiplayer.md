# 09. オンライン対戦設計

- 状態: Draft
- 目的 / 内容:
  - 別ネットワーク、別端末間でも 2 人対戦できるようにするための部屋制オンライン対戦設計を定義する。
  - 部屋番号、接続方式、同期単位、権威状態、再接続、勝敗確定、Undo/Redo、警戒表示の扱いを明確にする。
- 理由:
  - 現行実装はローカル状態のみを前提としており、別拠点プレイに必要なサーバ権威、順序保証、再接続復帰が未定義。
  - 先に同期契約を決めないと、UI 追加だけ先行しても状態破綻や巻き戻り競合が起きる。
- 依存:
  - `src/gameLogic.ts`
  - `src/hooks/useFiveDGomoku.ts`
  - `src/types.ts`
  - 将来追加するオンライン通信用サーバ実装
- 影響範囲:
  - ゲームモード
  - 状態管理
  - 接続 UI
  - Undo / Redo
  - 警戒表示
  - 対戦終了表示
- 受入条件:
  - ルーム作成、入室、着手同期、勝敗同期、再接続復帰の責務が曖昧でない。
  - クライアント権威ではなくサーバ権威で整合が取れている。
  - 現行のローカル / AI モードを壊さず追加できる構造になっている。
- テスト方法:
  - 型定義とプロトコル helper の単体テスト。
  - 将来実装時に 2 クライアント接続、切断再接続、同時着手競合、勝敗確定後の追加着手拒否を検証する。

## 1. 目的

部屋番号を共有して、別の場所にいる相手と同じ対局を同期しながら遊べるようにする。

ここで言う「同期」は以下を含む。

- 盤面
- 手番
- 勝敗
- カーソルや断面のような UI 状態のうち、共有が必要なもの
- 切断からの復帰

## 2. スコープ

初期スコープは以下とする。

- 2 人対戦
- 非公開ルーム
- ルーム番号による参加
- サーバ権威の着手確定
- 同期対象:
  - 盤面
  - 手番
  - 勝敗
  - 最終着手
  - 参加者状態
- 切断後の再接続復帰

初期スコープから外すもの:

- 観戦者
- マッチメイキング
- ランキング
- ボイス / チャット
- オンライン対戦中の自由 Undo / Redo

## 3. 推奨アーキテクチャ

### 3.1 権威モデル

サーバ権威を採用する。

理由:

- このゲームは `phase`、`streak`、`winInfo`、`history` の整合が重要。
- 同時着手や遅延到着があると、クライアント権威では巻き戻しが複雑になる。
- 勝敗確定後の追加着手拒否を一元化できる。

クライアントは以下だけを先行ローカル反映してよい。

- カーソル
- 断面表示
- カメラ
- ホバー
- ローカル警戒表示

着手結果そのものはサーバ確定後にのみ反映する。

### 3.2 通信方式

推奨は `WebSocket`。

理由:

- 対局は双方向で低遅延更新が必要。
- move ごとに polling すると遅いだけでなく競合処理が複雑になる。

補助的に HTTP を使う箇所:

- ルーム作成
- ルーム参加前の存在確認
- ヘルスチェック

## 4. ルーム番号

### 4.1 形式

- 長さ: 6 文字
- 文字集合: `23456789ABCDEFGHJKLMNPQRSTUVWXYZ`
- 表示形式: `ABC-DEF`
- 内部保存形式: `ABCDEF`

この形式にする理由:

- `0/O`、`1/I/L` の取り違えを避ける。
- 音声共有や手入力でも扱いやすい。
- 6 文字でも総数は十分多い。

### 4.2 生成

- ルーム作成時にサーバが発行する。
- クライアント側で候補を決め打ちしない。

### 4.3 入室

入室方法は 2 つ用意する。

- ルーム番号入力
- 招待リンク

招待リンク例:

```txt
https://example.com/?room=ABCDEF
```

## 5. 同期単位

### 5.1 サーバスナップショット

サーバはルームごとに以下の権威状態を保持する。

- `roomCode`
- `status`
- `settings`
- `board`
- `activePlayer`
- `revision`
- `moveNumber`
- `winInfo`
- `participants`
- `lastMove`
- `createdAt`
- `updatedAt`

### 5.2 revision

`revision` は着手確定ごとに `+1` する。

目的:

- 遅延パケット検出
- 競合同時着手の拒否
- 再接続後の差分判断

クライアントは着手時に `expectedRevision` を必ず送る。

## 6. 着手同期

### 6.1 基本フロー

1. クライアントが `submit_move` を送る
2. サーバが以下を検証する
   - ルームが存在する
   - 送信者が着手権を持つ
   - `expectedRevision` が現在 revision と一致
   - まだ勝敗確定していない
   - 座標が盤面範囲内
3. サーバが `makeMove` と `checkWin` を実行する
4. サーバが `revision`、`moveNumber`、`activePlayer`、`winInfo` を更新する
5. サーバが新 snapshot を両者へ送る

### 6.2 クライアント側の扱い

- 着手入力直後は「送信中」表示までに留める
- 確定石の更新はサーバ返答後
- 競合失敗時は snapshot を再取得して追従

## 7. 再接続と参加者管理

### 7.1 参加者

各プレイヤーには以下を持たせる。

- `seat`: `white` / `black`
- `playerToken`
- `displayName`
- `connected`
- `lastSeenAt`

### 7.2 再接続

再接続は `roomCode + playerToken` で行う。

要件:

- タブリロードで席を維持できる
- 一時切断後も一定時間は復帰できる
- 復帰時に最新 snapshot を再送する

### 7.3 タイムアウト

推奨値:

- 再接続猶予: 5 分
- 両者切断後のルーム保持: 10 分
- 完全放置ルーム削除: 24 時間

## 8. オンライン時の Undo / Redo

初期スコープではオンライン対戦中の `Undo / Redo` は無効化する。

理由:

- ローカル履歴とサーバ履歴が乖離しやすい
- 合意取り消し UI を設計しないまま有効化すると競合源になる
- 勝敗確定後の巻き戻しと再開の扱いが重い

将来拡張時の方針:

- `undoPolicy = mutual_consent`
- 一方が要求し、相手が承認したらサーバが巻き戻す
- `redo` はオンラインでは採用しない

## 9. 警戒表示とヒント

警戒判定そのものは deterministic なのでクライアントローカル計算でよい。

ただし以下はサーバ権威にしない。

- 警戒表示の ON/OFF
- グリッド補助表示
- 断面表示
- カメラ

理由:

- 対局結果に影響しない UI 状態まで同期対象にすると帯域と複雑性が増える
- 現行設計でもこれらは認知支援であり、ゲームの正解状態ではない

## 10. 勝敗確定後

勝敗確定後のサーバ挙動:

- 追加着手を拒否
- 最新盤面は保持
- クライアントは勝敗オーバーレイを閉じても盤面確認可能

クライアント要件:

- オーバーレイを閉じても観戦状態に留まる
- 新しい石は置けない
- 何で負けたか確認できる

## 12. 実装候補

プロトコルは provider 非依存で定義する。

実装候補の優先度:

1. Cloudflare Durable Objects
   - ルーム単位の状態保持に適している
   - WebSocket と部屋権威の相性がよい
2. Node.js + WebSocket サーバ
   - ローカル開発しやすい
   - 将来 Redis 併用で横展開可能

初期実装では 1 か 2 を選べばよく、フロント側の protocol / snapshot 契約は共通に保つ。

## 14. 受入チェック

- ルーム作成側が部屋番号を取得できる
- 別端末から部屋番号で参加できる
- 一方の着手が他方へ 1 revision として同期される
- 同時着手時に片方だけ採用される
- 勝敗確定後は双方で追加着手できない
- オーバーレイを閉じても勝敗確定盤面を確認できる
- リロード後に同じ席へ復帰できる
